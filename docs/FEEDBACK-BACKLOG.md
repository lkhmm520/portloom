## P0 — 路由协议与域名限制(反馈最集中)

### 1. HTTP / HTTPS 应拆分为独立选项,不强制 HTTPS
- **现状**:添加路由时协议只有一个 `http`(实际按 HTTPS 发布),无法选择"仅 HTTP 不上证书/不跳转"。
- **问题**:部分内网服务本身不支持或不需要 HTTPS;用户无法控制发布形态。
- **建议**:
  - 路由协议拆为 `http` / `https`(或增加"强制 HTTPS 跳转"开关);
  - 允许仅 HTTP 发布的域名跳过 autocert 申请。
- 对应反馈:#2、#6

### 2. 一个域名只能挂一条路由,浪费域名
- **现状**:`routes` 表对启用的 HTTP 域名有唯一性约束(`HTTPDomainEnabled`),同一域名添加第二条路由直接报错;公网域名也不能与面板管理域名相同。
- **问题**:一个服务占一个域名,域名成本高、网址难记。
- **建议**:
  - 支持**路径前缀路由**:同一域名下按 `/path` 分发到不同后端(如 `example.com/jellyfin`、`example.com/qb`);
  - 或支持同域名不同公网端口发布;
  - 评估放开"公网域名 = 管理域名"的限制(按路径与管理路由共存,需注意与 WebUI/API 路径冲突及安全隔离)。
- 对应反馈:#2、#4、#6

### 3. 协议支持扩展:真实的 TCP / UDP 转发
- **现状**:TCP 字段仅作为元数据保留,不会真正创建公网监听(README 已声明边界);UDP 完全不支持。
- **建议**:
  - 落地 TCP 公网发布(`tcpedge` 已有状态机雏形:published / conflict / bind_error 等,补齐端到端链路);
  - 评估 UDP 转发(注意:SSH 隧道原生不支持 UDP,需另走 UDP-over-TCP 封装或换传输层,成本较高,可先排期调研);
  - WebUI 协议下拉:`HTTP / HTTPS / TCP / UDP`。
- 对应反馈:#3、#6

## P1 — 部署灵活性

### 4. 管理入口 80/443 端口可配置
- **现状**:代码层已支持 `TM_EDGE_HTTP_ADDR` / `TM_EDGE_HTTPS_ADDR` 环境变量改端口,但安装脚本、compose 示例和文档均未暴露,WebUI 也无法修改。
- **注意**:改掉 80 端口后 ACME HTTP-01 验证会失效,需同时提供 DNS-01 或自定义证书方案,并在文档中明确说明。
- **建议**:安装器增加 `--http-port/--https-port` 参数;文档补充改端口 + 证书的组合说明。
- 对应反馈:#1

## P2 — 可观测性

### 5. 仪表盘增加监控图表
- **现状**:仪表盘只有路由/隧道状态,无资源与流量视图。
- **建议**:
  - 每条路由/每个 Agent 的流量统计(入/出字节数、请求数,gateway 与 tcpedge 侧埋点);
  - Server 与 Agent 的 CPU / 内存使用率上报与展示;
  - 前端为零依赖原生 JS,图表可用轻量方案(如内联 SVG/canvas)保持无构建。
- 对应反馈:#5
